# Consensus — Phase 3: UX Specification

Per-screen specs for the 12 screens inventoried in
[02-product-design.md](02-product-design.md). Global rules first; screens
reference them instead of repeating them.

> **Web translation note.** This spec was written iOS-first; per the platform
> amendment in docs/01 the product is a web app. The screen designs stand as
> written — this table maps the platform-specific concepts. The product is
> designed _mobile-first_ (it's used on phones in a group), desktop as a
> responsive bonus.
>
> | Spec says                | Web implementation                                                                                                                                                   |
> | ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
> | matchedGeometryEffect    | Motion (Framer Motion) `layoutId` shared-element transitions                                                                                                         |
> | Spring animations        | Motion springs (same response/damping house values)                                                                                                                  |
> | Haptics                  | `navigator.vibrate` where it exists (Android); elsewhere the visual/audio punctuation carries the beat alone — every haptic in this spec already pairs with a visual |
> | SF Pro / SF Rounded      | System font stack (`system-ui`); a rounded display face for numbers that move                                                                                        |
> | Dynamic Type             | `rem`-based type scale; respect browser font-size settings                                                                                                           |
> | VoiceOver custom actions | ARIA + full keyboard operability (vote buttons are real buttons; the card is reachable, not gesture-only)                                                            |
> | Reduce Motion            | `prefers-reduced-motion` media query, same substitutions                                                                                                             |
> | Share sheet              | Web Share API with copy-link fallback                                                                                                                                |
> | @AppStorage              | `localStorage`                                                                                                                                                       |
> | App relaunch re-entry    | Page reload re-entry — same server-state rule                                                                                                                        |

## Global design language

### Look

- **Cinematic, dark-leaning session flow.** Home/setup screens use system
  backgrounds and adapt to light/dark normally. Once a session starts (lobby
  onward), the backdrop is a dark, blurred gradient derived from the deck's
  poster colors — the "lights go down" moment. This is a deliberate mode
  shift: setup is a tool, the session is a game.
- **One accent color** (electric indigo family), semantic system colors for
  everything else. Vote colors: green (yes), red (no), pink (favorite) — used
  only as feedback flashes, never as static chrome.
- **Typography:** SF Pro throughout; SF Pro Rounded for numbers that move
  (scores, countdowns, "3 of 5"). Text is minimal by principle — if a screen
  needs a paragraph, the screen is wrong.

### Motion

- Spring animations everywhere (`.spring(response: 0.35, dampingFraction: 0.8)`
  as the house default); no linear/ease-in-out except crossfades.
- `matchedGeometryEffect` carries the _same poster_ through the app: voting
  card → leaderboard row → wheel segment → winner hero. The movie is a
  physical object that travels; it never pops in twice.
- **Reduce Motion:** every celebratory sequence has a crossfade equivalent —
  the wheel becomes a quick shuffle-highlight that settles on the winner,
  confetti becomes a glow pulse, the leaderboard build becomes a staggered fade.

### Haptics

Escalate with narrative stakes: light impact per vote, success notification on
favorite, rhythmic ticks during wheel deceleration, single heavy impact when
the winner lands. Silence everywhere else — haptics are punctuation, not noise.

### Accessibility

- Every gesture has a visible button twin (HIG: gestures are shortcuts, never
  the only path). Swiping the card and tapping 👎/👍 are the same action.
- Full Dynamic Type on setup screens; session screens use fixed-size poster
  art but scale all text and support VoiceOver custom actions
  ("Vote yes", "Vote no", "Favorite", "Show details") on the card.
- Vote feedback is never color-only: icon + motion + haptic accompany every
  color flash.

### Global states (apply to every session screen)

- **Connection lost:** thin banner "Reconnecting…" slides under the status
  bar; the screen stays interactive where possible (votes queue locally and
  flush on reconnect). After 30 s, a blocking sheet offers Retry / Leave.
- **Session ended by host / expired:** full-screen state with one action,
  "Back to Home".
- **Re-entry:** app relaunch during a session lands directly on the screen
  matching current server state (see Phase 2 state machine), with no
  intermediate navigation flashes.

## 1. Home

**Layout.** Large title "Consensus". Two stacked actions dominate the upper
half: **New Room** (filled, accent) and **Join Room** (bordered). Below, a
"Recent" section lists past sessions (winner poster thumbnail, category icon,
relative date) from local history.

**States.**

- _First run:_ Recent section replaced by a single friendly line —
  "Decisions you make together will show up here." No tutorial, no
  onboarding carousel; the app must be self-evident or it has failed.
- _Has history:_ up to 10 rows, tap shows a read-only winner recap.

**Animation.** Buttons have a subtle press-down scale (0.97). Nothing else
moves; Home is calm on purpose so the session feels alive by contrast.

## 2. Category chooser

**Layout.** Grid of large category cards. **Movies** is live (poster collage
background). Future categories (Restaurants, Games, …) appear as visibly
locked cards with a "Soon" badge.

**Why show locked cards:** it communicates the product's ambition, invites
feature requests, and costs one asset each. Tapping one gives a light shake +
haptic, nothing more.

**States.** None beyond default — this screen is static in MVP.

**Animation.** Selecting Movies zooms the card slightly toward the viewer,
then pushes to Filters.

## 3. Filters

**Layout.** Scrolling form, but styled as chips and cards — never `Form` rows
(survey energy). Sections top-to-bottom:

1. **Streaming services** — logo chips, multi-select, fetched from TMDb
   watch-providers for the user's region. First position, because it's the
   killer filter.
2. **Genres** — text chips, multi-select, empty selection = all.
3. **Era** — segmented: All · 2020s · 2010s · 2000s · Classics.
4. **Runtime** — toggle "Under 2 hours".
5. **Deck size** — 10 / 15 / 20 selector, default 15.

**States.**

- _Loading:_ service chips show shimmer placeholders.
- _Provider fetch failed:_ inline retry; user may proceed without the
  availability filter, with a one-line warning ("Results may not be
  streamable on your services").
- _No services selected:_ allowed, but a soft prompt appears at Review.

**Animation.** Chips toggle with a spring scale + accent fill. A live count
pill floats at the bottom: "~120 movies match" updating as filters change —
this is the feedback loop that teaches hosts to narrow.

## 4. Review & Create

**Layout.** Summary card (category, services as logos, genres, era, deck
size) with Edit buttons per row, then one full-width **Create Room** button.

**States.**

- _Creating:_ button shows inline spinner, controls lock.
- _Create failed:_ button shakes, inline error line, retry available; filters
  are preserved.
- _Thin results:_ if matches < deck size, warning with one-tap "Loosen
  filters" that clears the most restrictive one (computed, not guessed:
  drop the filter whose removal adds the most results).

**Animation.** On success, the summary card morphs (matched geometry) into
the Lobby's room-code card — creating a room _feels_ like the room appearing.

## 5. Join

**Layout.** A 6-character one-time-code style field (auto-uppercase,
auto-advance, paste-aware). `/join/{code}` URLs land here with the code
pre-filled and auto-submit — and since the lobby's QR encodes that URL, any
phone's native camera scans straight in. **No in-app scanner exists**; the
web platform deleted that screen's hardest part (camera permissions, reticle
UI, denied states).

**States.**

- _Not found:_ field shakes, "No room with that code."
- _Already started:_ "This room is mid-vote — ask the host to invite you to
  the next round."
- _Expired / full:_ equivalent single-line states.

**Animation.** Successful code entry fills the last character, pauses a beat,
then the whole field morphs into the Lobby header.

## 6. Name entry

**Layout.** Appears only if no display name is stored: single text field,
auto-generated avatar (initials on a deterministic color derived from a hash
of the name), one Continue button. Sheet, not a push — it's a gate, not a
destination.

**States.** Empty name disables Continue. That's it. The name persists to
`@AppStorage`; returning users never see this screen again (editable in
Settings).

## 7. Lobby

**Layout.** Room code as the hero element (large, rounded type), flanked by a
QR code and a Share button (share sheet with the universal link). Below, the
participant strip: avatars pop in live as people join. Bottom: host sees
**Start Voting** (disabled below 2 participants, with "Waiting for at least
one more…"), guests see "Waiting for {host} to start".

**States.**

- _Solo host:_ participant strip shows the host plus a pulsing empty slot —
  the visual asks for friends.
- _Guest view:_ no Start button; code/QR still visible so guests can relay it.
- _Participant joins/leaves:_ avatar springs in / fades out, with a subtle
  haptic tick for the host.
- _Room expired:_ global state.

**Animation.** Avatar pop-in is the lobby's heartbeat — spring scale from 0.5
with a slight overshoot. When the host taps Start, the whole lobby recedes
(scale down + fade) and the first card rises: lights down, game on.

## 8. Voting — the core screen

**Layout.** Full-bleed poster card, stacked with the next two cards peeking
behind (depth cue for progress and momentum). Overlaid on the card's bottom
gradient: title, year, runtime, genre tags, rating. Top edge: progress dots
(one per card, filling as voted) and a discreet Leave button. Bottom edge:
three buttons — 👎, ❤️, 👍 — mirroring the gestures.

**Interactions.**

- Swipe right / tap 👍 → yes. Swipe left / tap 👎 → no. Swipe up / tap ❤️ →
  favorite (counts as an emphatic yes; see scoring in Phase 1 doc).
- **Tap card to flip:** back face shows synopsis, cast line, and
  where-to-watch logos. Flip back with a tap; voting from the back face is
  allowed via buttons.
- **Single undo:** a small "↩︎ undo" chip appears for 3 seconds after each
  vote and rescinds only the most recent one. Cheap to build, removes
  vote anxiety, can't be abused to game the reveal.

**States.**

- _Deck loading:_ card-shaped shimmer; votes impossible until first card is in.
- _Poster failed to load:_ typographic fallback card (title large on
  gradient) — never a broken-image glyph.
- _Last card:_ after the final vote the stack is empty for a beat, then
  transitions to Waiting.

**Animation.** Cards exit differently per vote — yes: flies right with a
green flash and slight upward arc; no: drops left with a red flash and a
gravity feel; favorite: bursts upward with a pink pulse and a tiny heart
particle. The vote should feel _cast_, not submitted.

**Accessibility.** Buttons are the primary documented interface; VoiceOver
users get custom actions on the card and hear "Card 4 of 15, Dune: Part Two,
2024." Reduce Motion swaps exits for directional crossfades.

## 9. Waiting

**Layout.** Centered ring showing "3 of 5" (SF Rounded), surrounded by
participant avatars that gain checkmarks as each finishes. Ambient slow drift
of blurred posters in the background keeps the screen alive. Host only: a
**Reveal now** button appears after the first 60 s of no state change, with a
confirmation dialog ("2 people haven't finished — reveal anyway?").

**States.**

- _Progress:_ ring and avatars update live with a soft tick.
- _Someone left:_ their avatar fades, the denominator updates, a quiet
  toast names the change.
- _All done:_ auto-advances to Reveal for everyone simultaneously — the sync
  moment the backend exists for.

## 10. Reveal

**The payoff screen — this is the product.** Sequence, fully choreographed:

1. Background dims further; a 1-second beat of anticipation (haptic drumroll,
   three ascending ticks).
2. Leaderboard builds bottom-up — 5th place slides in first, then 4th…
   score bars fill as each lands. Poster thumbnails use matched geometry from
   the voting cards.
3. If any card got yes/favorite from _everyone_: **unanimous banner** drops
   with confetti burst — the celebrated bonus moment.
4. Then either:
   - _Clear winner:_ #1 row expands and morphs into the Winner screen.
   - _Tie at the top:_ tied rows detach and arrange into the **wheel**.

**Wheel (sub-state).** Tied posters become wedge segments. Host drags to
spin (flick velocity → spin momentum); everyone else watches the identical
spin live. Deceleration with tick haptics per segment boundary; the landing
segment blooms into the Winner screen.

**States.**

- _Runoff instead:_ when scores are too flat to rank (no card above a
  threshold), the leaderboard shows the top 3–5 and a "Final round!" banner;
  auto-advances everyone back to Voting with the mini-deck (max 2 runoffs,
  then the wheel decides — Phase 1 rule).
- _Reduce Motion:_ leaderboard staggered-fades in; wheel becomes a
  shuffle-highlight that settles.

## 11. Winner

**Layout.** Winner poster as hero (arrived via matched geometry), confetti
settling behind it. Below: title, year, runtime, rating, and **watch-provider
logos as tappable deep links** ("Watch on Netflix" opens the Netflix app/page
— the session's last step is pressing play). Footer: **Done** (accent) and
**Another round** (returns the room to Lobby with the same filters, winner
excluded from the next deck).

**States.**

- _No provider links (availability filter was skipped):_ providers row shows
  "Check your services" fallback.
- _Winner recap (from Home history):_ same layout, no confetti, no Another
  round.

## 12. Settings

**Layout.** Static list: display name (edits the stored name), About,
attribution ("Movie data from TMDb · Availability from JustWatch" — required
by TMDb terms, see Phase 1), version. Nothing else. Settings screens don't
earn features in a five-minute app.

## Open UX questions carried into Phase 4

1. Vote queueing offline (banner spec above) implies client-side vote state
   that reconciles with the server — affects the sync protocol design.
2. The simultaneous reveal requires server-authoritative timing (all clients
   flip on the same state change), and the shared wheel spin requires the
   host's spin parameters to be broadcast — both are architecture
   requirements, not UI details.
3. Poster color extraction for session backdrops (screen 7+) should happen
   off the main thread at deck-load time and be cached with the deck.
